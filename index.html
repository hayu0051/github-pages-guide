CONCEPT
GitHubの概念：プロジェクト＝リポジトリ

はい、「プロジェクト」と「リポジトリ」は1対1で対応すると考えてOKです。
GitHubでは1つのWebサイト・アプリのまとまりを「リポジトリ（Repository）」と呼びます。

あなたの感覚
プロジェクト
家庭菜園アシスタント
yokohama2026
次のプロジェクト…
=
GitHubの用語
リポジトリ（Repository）
my-garden-assistant
yokohama2026
新しいリポジトリ…
アカウント全体のイメージ：
hayu0051（GitHubアカウント）1つ ├── yokohama2026/ ← リポジトリ①（今回作成） │ └── yokohama_course_guide.html │ 🌐 https://hayu0051.github.io/yokohama2026/yokohama_course_guide.html │ ├── my-garden-assistant/ ← リポジトリ②（移行予定） │ └── index.html │ 🌐 https://hayu0051.github.io/my-garden-assistant/ │ └── 次のプロジェクト/ ← リポジトリ③（追加可能・無制限・無料） └── index.html
⚠️ 混乱しやすいポイント 右上「＋」メニューの各項目について

「＋」を押すと複数の項目が表示されますが、HTMLをWebで公開する目的では「New repository」だけを使います。
特に「New project」は名前が似ていますが、全く別の機能なので注意してください。

メニュー項目 何をするもの？ 今回使う？
New repository ファイルを置いてWebサイトを公開する ✅ これを使う
New project タスク管理ボード（ToDoリストのようなもの）
※Netlifyの「Project」とも全く別物		 ❌ 使わない
New codespace クラウド上でプログラムを書く開発環境 ❌ 不要
New gist コードのメモ・断片を共有する ❌ 不要
New organization 会社・チームのグループアカウントを作る ❌ 不要
New issue バグ報告・要望の管理票を作る ❌ 不要
Import repository 他サービスからリポジトリを取り込む ❌ 不要
「New project」≠ 新しいプロジェクト（サイト）
GitHubの「Project」はタスク管理ツールの名称です。Netlifyの「Project（＝サイト）」とも別物です。
新しいWebサイトを公開したいときは、必ず 「New repository」 を選んでください。
BACKGROUND
なぜGitHub Pagesを選んだか（Netlifyとの比較）

❌ Netlify（今回は断念）

クレジット制300/月（使い切ると全停止）
今回の状況クレジット0で全サイト停止
復活時期翌月（旅行に間に合わない）
AIエージェント使うとクレジット大量消費
停止の影響全プロジェクトが一斉停止

✅ GitHub Pages（採用）

クレジット制なし
突然の停止起きない
無料枠100GB/月・1GB/リポジトリ
有効期限無期限
リポジトリ数無制限
FIRST TIME SETUP
【初回】新規プロジェクトの作成〜公開（全5ステップ）

今回のyokohama2026で実際に行った手順の記録です。

1
GitHubにログイン
既存アカウント使用（hayu0051）
  1. https://github.com をブラウザで開く
  2. 右上の Sign in をクリック
  3. メールアドレスとパスワードを入力してログイン
  4. 二段階認証が求められた場合はメールのコードを入力
大昔に作ったアカウントでもOK。パスワードの控えがあればそのまま使える。
2
新しいリポジトリを作成
リポジトリ＝プロジェクト単位のフォルダ
  1. 右上の ＋ ボタン → New repository を選択
  2. 以下の通り設定する
リポジトリ名yokohama2026（半角英数字・ハイフン使用可）
説明（任意）横浜ドライブコース案内 2026年5月22日
可視性「Public（公開）」を選択（必須）
READMEを追加オンにする（チェックを入れる）
.gitignore / ライセンス「なし」のままでOK
  1. リポジトリを作成する をクリック
「Public」にしないとGitHub Pagesが使えない。必ず公開を選ぶこと。
3
HTMLファイルをアップロード
必ずindex.htmlという名前にしてからアップロード
  1. アップロード前に、PC上でHTMLファイルを「index.html」という名前に変更しておく
  2. リポジトリ画面で Add file → Upload files を選択
  3. 大きなグレーの枠に index.html をドラッグ＆ドロップ
  4. ファイル名が表示されたことを確認
  5. 変更をコミットする（Commit changes）をクリック
⚠️ なぜindex.htmlにしないといけないか
GitHubは /リポジトリ名/ のような短いURLにアクセスされた際、必ず「index.html」を探します。 別のファイル名（例：garden_app_v3.html）のままだと、index.htmlが見つからず代わりに README.mdの内容が表示されてしまいます。

ファイル名による動作の違い：
index.html の場合 → https://hayu0051.github.io/リポジトリ名/ で正常表示 ✅
別名（garden_app_v3.html等）の場合 → https://hayu0051.github.io/リポジトリ名/ はREADMEが表示される ❌
　→ https://hayu0051.github.io/リポジトリ名/garden_app_v3.html なら表示される ⚠️（URLが長くなる）
コメント欄（コミットメッセージ）は空欄でOK。
4
GitHub Pages を有効化
Settings → Pages から設定・URLが発行される
  1. リポジトリ上部の Settings をクリック
  2. 左メニューをスクロールして Pages をクリック
  3. 「Branch」の項目で None を main に変更
  4. Save をクリック
  5. 2〜3分そのまま待つ（ビルド処理中。焦らず待つこと）
  6. ページを再読み込みすると、上部に以下のように公開URLが表示される
✅ 成功するとこのように表示される
Your site is live at
https://hayu0051.github.io/リポジトリ名/
Last deployed by hayu0051 ○ minutes ago
右側の Visit site ボタンを押すと実際の表示を確認できる。
READMEを追加していた場合、Pages画面を開いた時点で既にURLが表示されていることがある（今回のyokohama2026はこのケース）。その場合はSave不要でそのままURLをコピーすればOK。
2〜3分待ってもURLが表示されない場合は、ページを再読み込み（F5）してもう少し待つ。 それでも表示されない場合はBranch設定が「main」になっているか再確認すること。
5
URLをLINEで送信
これで公開完了
公開URLの形式
https://ユーザー名.github.io/リポジトリ名/ファイル名.html
今回の実際のURL
https://hayu0051.github.io/yokohama2026/yokohama_course_guide.html
  1. Pages画面の Visit site で表示を確認
  2. URLをコピーしてLINEに貼り付けて送信
  3. 相手はURLをタップするだけで即表示
OPERATION ①
【操作①】既存ファイルを更新する方法

例：yokohama_course_guide.html の内容を修正して再アップロードしたい場合

🔄 方法A：ファイルを上書きアップロード（最も簡単・推奨）
  1. https://github.com/hayu0051/yokohama2026 を開く
  2. Add file → Upload files をクリック
  3. 更新したHTMLファイル（同じファイル名） をドラッグ＆ドロップ
  4. 変更をコミットする をクリック
  5. 1〜2分後にURLが自動更新される（URLは変わらない）
同じファイル名でアップロードすると自動的に上書きされる。URLはそのままなのでLINEで送り直す必要はない。
更新が反映されていないように見える場合は Ctrl+Shift+R でスーパーリロード。
✏️ 方法B：GitHub上で直接編集（小さな修正に便利）
  1. リポジトリ画面でファイル名（例：yokohama_course_guide.html）をクリック
  2. 右上の鉛筆アイコン ✏️ Edit this file をクリック
  3. テキストエディタが開くので直接編集する
  4. 右上の Commit changes → Commit changes で確定
  5. 1〜2分後にURLが自動更新される
HTMLコードを直接編集するため間違いが起きやすい。Claudeで修正済みのHTMLを方法Aで上書きする方が確実でミスが少ない。
OPERATION ②
【操作②】新しいプロジェクトを追加する方法

例：家庭菜園アシスタントをNetlifyからGitHub Pagesに移行する場合。初回と同じ手順で、リポジトリを新しく作るだけ。

✅ 手順（初回と同じ。Step2のリポジトリ名だけ変える）
  1. GitHubにログインした状態で右上 ＋ → New repository
  2. リポジトリ名を入力（例：my-garden-assistant）
  3. Public を選択・READMEを追加 にチェック → 作成
  4. HTMLファイルをドラッグ＆ドロップでアップロード → コミット
  5. Settings → Pages → Branch: main → Save
  6. URLが発行される
家庭菜園アシスタントのURLになる例（index.htmlの場合）
https://hayu0051.github.io/my-garden-assistant/
GAS・Gemini API連携はそのまま動く。
GASはGoogleのサーバーで動くため、HTMLをどこに置いても影響なし。NetlifyのHTMLをダウンロードしてGitHubに上げるだけで移行完了。
ファイル名を index.html にするとURLが短くなる：
index.html の場合 → https://hayu0051.github.io/my-garden-assistant/（末尾のファイル名不要）

🌐 公開中のPages URLを確認する
  1. 確認したいリポジトリを開く
  2. Settings → Pages を開く
  3. 「Your site is live at…」に公開URLが表示される
  4. Visit site ボタンで動作確認できる
リポジトリのトップ画面（Codeタブ）の右側「About」欄の歯車アイコン → 「Website」欄にURLを貼り付けておくと、一覧から直接URLが確認できて便利。
🗑 リポジトリ（プロジェクト）を削除する
  1. 削除したいリポジトリを開く
  2. Settings → 画面の一番下「Danger Zone」セクションへスクロール
  3. Delete this repository をクリック
  4. 確認のためリポジトリ名（例：yokohama2026）を入力
  5. I want to delete this repository をクリック
削除すると元に戻せない。URLも即座に無効になる。旅行終了後に不要になったら削除してOK。

TROUBLESHOOTING
トラブルシュート
❌ Netlifyで「サイトにアクセスできません」エラー
原因：Netlifyのクレジット（300/月）を使い切り、アカウント全体が停止。
確認：Netlify管理画面 → Usage & billing → Credits available: 0
対処：翌月リセットまで待つか、GitHub Pagesに切り替える。
根本原因：「Build with an AI agent（Claude Agent）」を使うとクレジットを大量消費する。
❌ HTMLにiframeでGoogleマップを埋め込んだらCSPエラー
原因：Claudeのプレビュー環境がiframeによる外部サイト読み込みをブロック（net::ERR_BLOCKED_BY_CSP）。
対処：iframeを削除し、「地図を開く」リンクと「Googleマップでルート確認」ボタンに置き換える。
補足：GitHub Pages上で開けばiframeも表示可能。
❓ ファイルを更新したのにURLが古いまま
原因：ブラウザがキャッシュした古いページを表示している。
対処：Ctrl+Shift+R（Windows）または Cmd+Shift+R（Mac）でスーパーリロード。またはアップロードから1〜2分待ってから確認する。
❓ Pages URLにアクセスすると404エラー
原因1：Settings→Pagesで「Branch: main」に設定して「Save」していない。
原因2：URLのファイル名が違う（大文字・小文字・拡張子に注意）。
原因3：ビルドがまだ完了していない（1〜2分待つ）。
FAQ
FAQ
Q. ファイル名はindex.htmlにしないといけない？
A. LINEで送る場合はどちらでもOK。
index.html にするとURLが短くなる（末尾のファイル名が不要）。
index.html → https://hayu0051.github.io/yokohama2026/
yokohama_course_guide.html → https://hayu0051.github.io/yokohama2026/yokohama_course_guide.html
Q. セキュリティは大丈夫？
A. 個人情報が含まれていなければ実質問題なし。
URLを知っている人なら誰でも見られる（無料プランではパスワード保護不可）。
検索エンジンに引っかかる可能性は低く、旅行プランHTMLであればリスクはほぼゼロ。
旅行後にリポジトリを削除すれば完全に消える。
Q. 家庭菜園アシスタント（GAS・Gemini API使用）もGitHub Pagesに移せる？
A. 移せる。GAS側は一切変更不要。
GASはGoogleのサーバーで動くため、HTMLをどこに置いても影響なし。
NetlifyにあるHTMLファイルをダウンロードし、新しいGitHubリポジトリにアップロードするだけ。Netlifyのクレジット切れで突然停止する心配がなくなる。
Q. 無料でいくつリポジトリ（プロジェクト）を作れる？
A. Publicリポジトリは無制限。
GitHubの無料プランではPublicリポジトリを無制限に作成でき、GitHub Pagesも各リポジトリに1つ有効化できる。実質プロジェクト数分のサイトを無料・無制限で公開可能。
